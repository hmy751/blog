# Article Types

글 유형은 분류표가 아니다. 작업 초기에 어떤 재료를 찾고, 어떤 렌즈를 우선 켤지 정하는 시작점이다.

이 문서는 톤을 정하지 않는다. 톤은 [voice.md](../lenses/voice.md)가 맡고, 단계는 [workflow.md](workflow.md)가 맡는다. 여기서는 "이 유형의 글이 좋은 글이 되려면 어떤 글감이 살아 있어야 하는가"만 본다.

## 공통 Material Signature

모든 글은 아래 중 일부를 가진다.

- 중심 질문: 이 글을 쓰게 만든 문제나 궁금증.
- 구체 장면: 실행, 실패, 대화, 수치, 화면, 코드 구조, 반복된 경험.
- 불일치: 예상과 결과, 말과 실행, 기록과 제어, 현상과 원인 사이의 어긋남.
- 판단 기준: 무엇을 문제로 봤고, 어떤 제약과 기준으로 선택했는가.
- 변화: 처음 이해, 가정, 선택 기준이 어떻게 바뀌었는가.
- 근거: 독자가 판단을 복구할 수 있는 수치, 표, 코드 예시, 도식, 스크린샷, 짧은 사례.
- 다음 기준: 이 경험이 다음 실험, 구현, 글쓰기, 판단에 무엇을 남겼는가.

모든 글이 이 항목을 전부 채울 필요는 없다. 현재 글 유형과 초안의 중심 질문에 필요한 항목만 본다.

## Type Map

| Type | 주요 재료 | 우선 렌즈 |
| --- | --- | --- |
| `technical-case-study` | 문제, 제약, 대안, 구현 선택, 트레이드오프, 결과 해석 | `developer`, `reader-flow`, 필요 시 `supporting-materials` |
| `company-project` | 공개 가능한 문제, 조직/서비스 제약, 개선 선택, 효과와 한계 | `source-policy`, `developer`, `portfolio-signal` |
| `product-architecture` | 사용자 문제, 제품 제약, 책임/상태/데이터 흐름, 설계 결정 | `developer`, `reader-flow`, `portfolio-signal` |
| `retrospective` | 반복된 경험, 변화의 축, 자기 관찰, 현재 상태, 다음 방향 | `voice`, 필요 시 `edit-patterns` |
| `learning-experiment` | 가설, 실험 조건, 측정, 오독, 재해석, 다음 질문 | `voice`, `developer`, 필요 시 `supporting-materials` |

## Technical Case Study

대상:

- 구현, 트러블슈팅, 기술 도입, 비교, 아키텍처 복기.

Material signature:

- 글쓴이가 실제로 문제로 본 현상.
- 그 문제가 단순 버그, 구조 문제, 사용자 흐름 문제, 운영 문제 중 어디에 가까웠는지.
- 선택 가능한 대안과 버린 대안.
- 선택 기준: 안정성, 비용, 속도, 유지보수성, 설명 가능성, 사용자 경험.
- 구현이나 디버깅 과정에서 판단이 바뀐 지점.
- 독자가 볼 수 있는 코드 구조, 흐름, 수치, 표, 화면, 전후 비교.
- 남은 한계와 다음에 다르게 볼 기준.

약한 신호:

- "무엇을 했다"만 있고 왜 그게 문제였는지 없다.
- 기술 설명은 많은데 선택 기준과 버린 대안이 없다.
- 결과 수치가 있는데 수치를 어떻게 읽었는지 없다.

## Company Project

대상:

- 회사나 서비스에서 수행한 작업의 공개 가능한 복기.

Material signature:

- 공개 가능한 수준으로 일반화한 문제 상황.
- 서비스, 팀, 일정, 안정성, 사용자 영향 같은 제약.
- 개선 전후의 차이. 단, 비공개 수치나 내부 프로세스는 숨긴다.
- 협업 맥락은 역할과 결정 기준 중심으로만 남긴다.
- 효과는 과장하지 않고 확인 가능한 범위로 말한다.
- 회사 내부 정보 없이도 독자가 판단할 수 있는 구조, 흐름, 예시.

약한 신호:

- 회사 내부 사정이나 담당자 평가가 글의 근거처럼 올라온다.
- 공개할 수 없는 수치를 일반화하다가 의미가 바뀐다.
- 자기소개서처럼 성과만 남고 문제 정의가 없다.

## Product Architecture

대상:

- 개인/제품 프로젝트의 문제 정의와 설계 결정.

Material signature:

- 실제 사용자 불편이나 제품 흐름에서 시작한 질문.
- 상태, 책임, 데이터 흐름, 오프라인/동기화, 권한, 유지보수성 같은 설계 압력.
- 설계 후보와 선택 기준.
- 선택한 구조가 어떤 행동이나 운영을 가능하게 했는지.
- 설계의 비용, 한계, 나중에 바꾸고 싶은 부분.
- 독자가 따라갈 수 있는 구조도, 상태 흐름, before/after, 짧은 예시.

약한 신호:

- 제품 문제보다 기술 선택의 장점만 앞선다.
- 구조 설명은 있는데 사용자가 왜 영향을 받는지 없다.
- 아키텍처 도식이 장식이고 판단을 돕지 않는다.

## Retrospective

대상:

- 개인 회고, 블로그 회고, 커리어/학습 방향 회고, 글쓰기 자체를 돌아보는 글.

Material signature:

- 한 번의 사건보다 반복해서 돌아온 질문이나 감각.
- 변화를 만든 실제 장면이나 반복된 경험.
- 그때의 생각과 지금의 판단 사이의 차이.
- 과장된 다짐이 아니라 현재 상태와 다음 행동.
- 기술 글의 증명 방식이 없어도 서는 중심 문장.

약한 신호:

- 변화가 한 문장 선언으로만 나온다.
- 자기 관찰이 사실 범위 없이 분위기만 만든다.
- 앞으로의 방향이 구체 행동 없이 다짐으로 닫힌다.

## Learning / Experiment

대상:

- PI Lab 글, 기술 학습 기록, 실험, 측정, 실패, 재해석 중심의 글.

Material signature:

- 처음의 가설, 기대, 또는 잘못 읽은 지점.
- 실험 조건: 데이터, 모델, 파라미터, 평가 기준, 비교 대상.
- 측정값과 관측 결과.
- 예상과 결과가 어긋난 지점.
- 오독, 재해석, 판단이 바뀐 순간.
- 이해, 가정, 비교, 선택 기준이 실제 실험 설계, 지표 해석, 디버깅, 다음 행동으로 이어진 흔적.
- 독자가 직접 판단을 복구할 수 있는 표, 그래프, trace, 실행 결과, 코드 구조, 스크린샷 후보.
- 다음 실험이나 다음 글로 이어지는 질문.

약한 신호:

- 실험 로그는 있는데 무엇을 잘못 봤고 어떻게 다시 보게 되었는지 없다.
- 개념 설명이 뒤의 실험 판단과 연결되지 않고 튜토리얼처럼 떠 있다.
- 수치가 있는데 실험 조건이나 해석이 붙어 있지 않다.
- "많이 배웠다"로 닫히고 다음 기준이 남지 않는다.

PI Lab 글:

- 먼저 `learning-experiment`로 본다.
- PI Lab이라는 이름이 구조, 도입, 마무리, 개인 맥락을 자동으로 정하지 않는다.

## 단계별 사용

Material:

- 해당 type의 material signature를 보고 원천 자료에서 빠진 장면, 질문, 오독, 선택 기준을 찾는다.

Shaping:

- 찾은 재료를 결과 나열이 아니라 판단의 흐름으로 배열한다.
- artifact gap이 보이면 [output-contracts.md](output-contracts.md)와 [supporting-materials.md](../lenses/supporting-materials.md)를 참고한다.

Texture:

- polish 과정에서 해당 type의 핵심 재료가 사라지지 않았는지 본다.
- 다듬은 뒤 글이 더 읽히지만 더 generic해졌다면, 어떤 재료가 평평해졌는지 확인한다.

Prepublish:

- 글 유형이 분명한지 확인하되, 이 문서를 hard guard처럼 적용하지 않는다.
- 공개 경계와 frontmatter는 [prepublish-check.md](../guards/prepublish-check.md)가 맡는다.

## 멈춤 신호

- article type을 고정 템플릿처럼 적용해 현재 초안의 좋은 질문을 눌러버린다.
- 한 글이 여러 유형을 섞고 있는데 주된 질문을 정하지 못해 모든 렌즈를 한 번에 켠다.
- `learning-experiment`를 PI Lab 전용 구조로 오해한다.
- 회고 글에 기술 글의 증명 방식을 강제한다.
- 기술 글에 개인 서사나 포트폴리오 신호를 강제로 삽입한다.
