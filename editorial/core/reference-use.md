# Reference Use

레퍼런스는 active rule이 아니라 pattern source다.

하네스에는 레퍼런스 파일명, 플랫폼명, 특정 글 제목, 표면 구성을 직접 박지 않는다. 대신 레퍼런스가 만든 효과를 현재 글과 다른 작업에서도 재사용 가능한 패턴으로 번역한다.

## 1. Observe

먼저 레퍼런스에서 실제로 작동한 구체 요소를 관찰한다.

예:

- 초반에 글의 질문과 읽는 관점을 빨리 잡는다.
- 긴 산문 중간에 코드, 표, 화면, 캡션, 짧은 리스트가 독자 위치를 회복시킨다.
- 자료 뒤에 긴 재설명 대신 다음 판단 한 문장이 온다.
- 화면으로 봐야 이해되는 재료를 산문이나 표만으로 축소하지 않는다.

## 2. Abstract

관찰한 요소가 왜 작동했는지 원칙으로 바꾼다.

좋은 변환:

- 초반 3~5문단 안에 글의 질문과 읽는 관점을 잡는다.
- 자료는 `claim -> artifact -> recovery` 흐름으로 배치한다.
- 화면으로 판단해야 하는 재료는 표로만 축소하지 않는다.
- 긴 산문은 독자가 판단을 복구할 수 있는 손잡이로 끊는다.

나쁜 변환:

- "제공된 캡처를 참고하면..."
- "특정 플랫폼 글처럼..."
- "레퍼런스에 표가 있으니 표를 넣는다."
- "이 글 유형에는 항상 히어로 이미지를 넣는다."

## 3. Adapt

추출한 패턴을 현재 글에 맞는 edit move로 바꾼다.

예:

- 관측성 글에서는 trace UI 후보 슬롯, QA pipeline 도식, config snapshot 표로 옮긴다.
- 평가 글에서는 metric 변화 표와 판단 회수 문장으로 옮긴다.
- 회고 글에서는 대표 장면과 판단이 바뀐 문단을 앞으로 당긴다.

레퍼런스에서 관찰한 장치를 그대로 복제하지 않는다. 현재 글의 중심 질문과 공개 경계에 맞는 가장 작은 move를 고른다.

## 4. Guard

옮기면 안 되는 표면 디테일을 명시한다.

- 특정 파일이나 캡처가 없어도 기준이 작동해야 한다.
- 디자인, 레이아웃, 이미지 수, 소제목 개수는 복제하지 않는다.
- 모든 글에 자료 장치를 강제하지 않는다.
- 공개 경계가 불확실한 화면이나 원문은 active guide로 승격하지 않는다.

## Reference Hardcoding Test

새 기준에서 레퍼런스 파일명, 플랫폼명, 특정 글 제목을 지워도 작동하는가?

- 작동하면 active guide나 output contract에 둘 수 있다.
- 작동하지 않으면 active guide에 두지 않는다. decision record나 audit에는 증거로 남길 수 있고, reference profile에는 다시 추출한 재사용 가능한 패턴만 둔다.

하네스 파일을 실제 수정한 뒤 레퍼런스 하드코딩이나 층위 드리프트가 의심되면 `blog-harness-observer`로 사후 점검한다. observer는 이 원칙을 다시 쓰는 장소가 아니라 준수 여부를 묻는 감사자다.
