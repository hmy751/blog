# Supporting Materials

자료 장치는 블로그답게 보이기 위한 장식이 아니라 독자의 판단을 돕는 보조 재료다. 표, 코드, 이미지, 도식, 그래프, 스크린샷은 본문으로 설명하면 길어지는 관계, 비교, 흐름, 전후 차이를 짧게 보여줄 수 있을 때만 쓴다.

이 문서는 새 글쓰기 단계가 아니다. `reader-flow-lens.md`를 보다가 실제 자료 장치 판단이 필요해졌을 때 추가로 여는 보조 기준이다.

## 기본 원칙

- 자료 장치는 글감을 대체하지 않는다.
- 자료 장치를 넣기 전에 보호해야 할 문장, 장면, 판단 변화를 먼저 확인한다.
- 자료가 들어가도 본문에는 "왜 보는가"와 "보고 나서 무엇을 판단했는가"가 남아야 한다.
- 공개 글에는 내부 경로, 비공개 코드, 개인 메모, 원천 자료 화면, 원천 문장을 노출하지 않는다.
- 모바일과 발행 화면에서 깨질 정도로 넓거나 복잡한 자료는 줄이거나 다른 형태로 바꾼다.
- 장치가 글을 더 읽히게 만들었는지와 동시에 더 평평하게 만들었는지 `Texture Keeper` 관점으로 다시 본다.

## 자료 앞뒤 문장

표, 코드, 이미지, 도식은 혼자 이해되면 좋은 것이 아니라 본문 판단과 연결될 때 힘이 생긴다. 자료를 넣을 때는 아래 3단계를 기본으로 둔다.

1. **열기**: 이 자료에서 무엇을 봐야 하는지 먼저 말한다.
2. **자료**: 표, 코드, 이미지, 도식, 그래프, 스크린샷을 보여준다.
3. **회수**: 자료를 보고 판단이 어떻게 바뀌었는지 붙인다.

예:

```text
숫자를 비교하면 top-k를 줄였을 때 단순히 Context 양만 줄어든 것이 아니었습니다.

[비교표]

이 표에서 중요한 것은 정확도보다 근거성이 더 크게 움직였다는 점입니다.
```

자료 앞뒤 문장이 없으면 독자는 자료를 "봐야 할 증거"가 아니라 "중간에 끼어든 이미지"로 읽기 쉽다.

## 장치별 기준

- **표**: 여러 항목이 3개 이상의 속성을 가질 때 적합하다. 예: `조정한 것 / 처음 기대 / 실제 관찰 / 남은 기준`.
- **리스트**: 순서가 없거나 단순 항목을 빠르게 보여줄 때 적합하다.
- **번호 리스트**: 실행 순서, 판단 순서, 원인 추적 순서가 중요할 때 적합하다.
- **코드 예시/의사코드**: 로직의 책임, 입력/출력, 프롬프트 구조, 평가 기준을 독자가 봐야 판단할 수 있을 때 적합하다. 내부 코드 원문을 공개할 수 없으면 공개 가능한 의사코드나 축약 예시로 바꾼다.
- **도식**: 파이프라인, 책임 분리, 데이터 흐름, 원인-결과 경로를 보여줄 때 적합하다.
- **그래프**: 숫자의 크기보다 변화 방향이나 비교 폭이 핵심일 때 적합하다.
- **스크린샷**: 실제 출력, UI 상태, 모델 답변 차이를 독자가 직접 봐야 판단할 수 있을 때 적합하다.
- **이미지/일러스트**: 분위기만 만드는 경우에는 우선순위가 낮다. 글의 핵심 질문이나 증거를 선명하게 할 때만 쓴다.
- **콜아웃/요약 박스**: 운영상 주의점, 실험에서 남은 규칙, 독자가 기억해야 할 제약을 잠깐 멈춰 세울 때 적합하다. Markdown에서는 blockquote, 짧은 번호 리스트, 굵은 문장+리스트로 대체할 수 있다.

## 사용 전 질문

- 이 자료가 없으면 독자가 관계나 차이를 놓치기 쉬운가.
- 본문 2~3문단을 표나 도식 하나로 줄일 수 있는가.
- 시각 자료가 글의 핵심 질문을 선명하게 하는가, 아니면 단순히 있어 보이게 하는가.
- 자료 앞문장에 볼 이유가 있고, 뒷문장에 해석이나 판단 회수가 있는가.
- 모바일에서 봐도 읽히는가.
- 발행 플랫폼과 모바일 화면에서 자료가 너무 넓게 깨지지 않는가. 읽기 어려운 긴 표라면 리스트, 짧은 표, 이미지화된 도식, 본문 분할 중 하나를 검토한다.
- 내부 경로, 비공개 코드, 개인 메모, 원천 자료 화면이 노출되지 않는가.
- 캡션이나 본문 연결 없이도 무엇을 봐야 하는지 알 수 있는가.

## 숫자와 비교 자료

숫자는 자랑이나 장식이 아니라 판단의 근거다. 숫자를 쓸 때는 아래 순서를 우선한다.

1. 무엇을 확인하려는 실험인지 먼저 둔다.
2. 조건과 비교 대상을 필요한 만큼만 밝힌다.
3. 숫자를 보여준다.
4. 그 숫자를 어떻게 읽었는지 바로 붙인다.
5. 다음 판단이나 남은 한계로 연결한다.

숫자가 많은 글에서는 산문 안에 모든 값을 묻지 말고, 독자가 비교해야 하는 값만 표나 짧은 리스트로 꺼낸다.

## 후보 표시

초안 편집 중 표, 코드, 이미지, 도식, 그래프, 스크린샷이 필요해 보이면 먼저 보호할 문장과 판단 변화를 확인한 뒤, 어떤 장치가 독자 판단을 가장 잘 돕는지 고른다. 직접 만들 수 있는 장치라도 글의 목소리나 발견을 평평하게 만들 위험이 있으면 바로 넣지 않고 후보로 제안한다.

필요한 자료나 공개 경계 판단이 남아 있을 때는 해당 위치에 주석으로 후보 슬롯을 남길 수 있다.

```md
<!-- supporting-material candidate
type: table | list | code | pseudocode | diagram | graph | screenshot | image
purpose: 독자가 무엇을 더 빨리 판단하게 할 것인가
placement: 어느 문단 뒤에 둘 것인가
source: 필요한 데이터, 출력, 화면, 실험 결과
style: 간단한 Markdown 표, Mermaid 도식, 공개 가능한 의사코드, 실제 스크린샷, 생성 이미지 등
user-action: 사용자가 제공해야 할 자료나 결정할 스타일
caption: 본문 아래에 붙일 설명
alt: 이미지 없이도 전달되어야 할 내용
public-boundary: 내부 경로, 비공개 코드, 개인 메모, 원천 문장 노출 위험
-->
```

이 주석은 발행 전에는 실제 자료로 바꾸거나 제거한다. 주석 자체가 공개 원고에 남아서는 안 된다.

## 멈춤 신호

- 자료가 본문 판단을 돕지 않고 분위기만 만든다.
- 표를 넣었지만 표 밖에 해석이 없다.
- 표, 코드, 도식, 스크린샷 앞뒤에 볼 이유와 회수 문장이 없다.
- 이미지나 도식이 본문 주장을 대신하고, 본문에서는 무엇을 봐야 하는지 말하지 않는다.
- 자료 장치를 넣으면서 살아 있는 장면, 실패, 판단 변화를 지웠다.
- 긴 표나 복잡한 도식이 모바일에서 읽히지 않는다.
- 시각 자료에 내부 경로, 비공개 코드, 개인 메모, 원천 자료 원문이 노출된다.
