# Prepublish Check

발행 전 마지막 확인 목록이다. `content/posts/`로 승격하기 전에 적용한다.

## Frontmatter

- `title`이 독자가 보는 제목과 일치한다.
- `date`가 실제 발행일이며 `TBD`가 아니다.
- `author`, `readTime`, `platform`, `tags`가 있고, `platform` 값은 `Blog`다.
- 글의 맥락에 따라 `project`, `company`/`service`, `category` 중 필요한 필드가 있다.
- 임시 상태값, 작업 메모, 내부 태그가 남아 있지 않다.

## 공개 경계

- 로컬 절대 경로가 없다.
- 내부 미션 코드 경로가 없다.
- PI Lab 코드 원문이나 내부 파일명이 노출되지 않는다.
- 면접 피드백, 개인 평가, private memo 문장을 그대로 쓰지 않았다.
- Claude/Codex 생성 흔적이 없다.
- `supporting-material candidate` 주석이 남아 있지 않다. 후보 슬롯은 편집 단계 장치이며 공개 원고에는 실제 자료로 바꾸거나 제거한다.

## 글 품질

- 글 유형이 분명하다: 기술 구현/복기, 회사 프로젝트, 제품/아키텍처, 개인 회고, 학습/실험 중 어디에 가까운지 확인한다.
- bold나 결론 문장이 본문에서 구체 근거를 받는다.
- 도입부가 실제 시간/논리 순서와 어긋나지 않는다.
- 같은 메타 반성이 두 문단 이상 반복되지 않는다.
- 담백함을 만들려고 장면을 과하게 압축하지 않았다.
- 수치, 용어, 시점, 사람 이름이 원천 자료와 맞는다.

## 글 유형별 추가 확인

- 기술 구현/복기 글: 문제, 제약, 대안, 선택, 구현, 한계가 분리되어 있다.
- 회사 프로젝트 글: 회사/서비스 맥락은 공개 가능한 수준으로 일반화되어 있다.
- 제품/아키텍처 글: 사용자 문제와 설계 트레이드오프가 함께 보인다.
- 개인 회고 글: 자기 관찰은 허용하되 과장된 자기 포장이나 근거 없는 전환으로 흐르지 않는다.
- 학습/실험 글: 실험 조건, 측정, 오독, 재해석, 다음 질문이 구분된다.

## 실행

```sh
node scripts/blog-prepublish-check.mjs
```

이 스크립트는 deterministic guard다. 통과해도 글이 좋은 것은 아니며, 실패하면 공개 경계나 발행 메타데이터를 먼저 고친다.
