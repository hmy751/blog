# Developer Lens

이 블로그는 기술 블로그다. 좋은 기술 글은 기술 설명이나 실험 결과만 남기는 글이 아니라, 글쓴이가 무엇을 문제로 봤고 어떤 기준으로 판단했는지가 보이는 글이다.

이 렌즈는 새 agent나 별도 단계가 아니다. `Material Partner`, `Shaping Editor`, `Texture Keeper`가 공통으로 참고하고, 필요하면 `Structure Critic`이 보조로 확인하는 관점이다.

## 핵심 질문

기술 글을 볼 때 아래 질문을 확인한다.

- 이 문제는 왜 글쓴이에게 걸렸는가.
- 무엇을 문제로 정의했고, 무엇은 문제로 보지 않았는가.
- 어떤 제약이 있었는가: 사용자, 서비스, 팀, 시간, 데이터, 모델, 인프라, 도구.
- 어떤 선택지를 고려했고, 무엇을 버렸는가.
- 선택의 기준은 무엇이었는가: 안정성, 비용, 속도, 유지보수성, 설명 가능성, 사용자 경험, 실험 해석 가능성.
- 사건과 결론 사이에서 어떤 이해, 가정, 비교, 선택 기준을 거쳐 판단했는가.
- 결과를 보고 처음 판단이 어떻게 바뀌었는가.
- 이 경험이 다음 개발 방식이나 판단 기준에 무엇을 남겼는가.

## 좋은 신호

- 문제 현상과 문제 원인을 분리해서 본다.
- 기술 선택이 "좋아서"가 아니라 제약과 목적 안에서 선택된다.
- 수치, 코드, 표가 자랑이나 장식이 아니라 판단의 근거로 쓰인다.
- 실패나 오독이 숨겨지지 않고 다음 판단으로 이어진다.
- 개념 설명이나 개인적 생각이 일반론으로 떠 있지 않고, 실제 선택·실험 설계·지표 해석·디버깅 기준으로 이어진다.
- FE 개발자로서의 기존 감각이 지워지지 않는다. 상태, 책임, 데이터 흐름, 사용자 흐름, 디버깅, 유지보수성 같은 기준이 AI/ML 글에서도 다른 언어로 이어질 수 있다.
- 독자가 읽고 "이 사람은 문제를 이렇게 쪼개고 판단하는구나"를 느낄 수 있다.

## 적용 예시

- 제품/아키텍처 글: 사용자 불편 → 제약 → 대안 → 설계 선택 → 트레이드오프 → 이후 개선 여지.
- 회사 프로젝트 복기: 기존 구조의 문제 → 공개 가능한 제약 → 점진적 개선 선택 → 개발 생산성/서비스 품질 변화.
- 학습/실험 글: 처음 가설 → 실험 조건 → 오독 → 측정으로 드러난 반례 → 다음 실험 기준.
- PI Lab 글: AI 입문기가 아니라, 기존 개발자의 문제 정의 감각이 AI/ML 실험의 언어로 번역되는 과정.

## 하지 않을 것

- 모든 글에 자기소개나 커리어 서사를 강제로 넣지 않는다.
- "개발자로서"라는 표현을 문장에 직접 넣으라고 요구하지 않는다.
- 문제 해결 욕구를 자기 포장이나 이력서 문체로 바꾸지 않는다.
- 단순한 기술 튜토리얼 글에 억지로 큰 서사를 만들지 않는다.
- 어그로성 반전이나 과장된 위기감을 만들지 않는다.

## 파트너별 사용

### Material Partner

아직 글에 드러나지 않은 재료를 찾는다.

- 왜 이 문제가 글쓴이에게 중요했는가.
- 어떤 실제 불편, 제약, 막힘, 선택의 압력이 있었는가.
- 글에는 결과만 있지만, 그 뒤에 판단이 바뀐 장면이 있는가.
- 결과 뒤에 숨어 있는 이해, 초기 가정, 비교, 선택 기준이 있는가.
- 학습/실험 글이라면 학습노트나 작업 메모에 개념을 자기 언어로 다시 잡은 흔적이 있는가.
- FE/제품/협업 관점이 일반 기술 설명 뒤에 숨어 있지 않은가.

### Shaping Editor

재료를 글의 흐름으로 배열한다.

- 문제 정의 → 제약 → 선택지 → 버린 대안 → 트레이드오프 → 바뀐 판단이 읽히는가.
- 기술 설명이 결과 나열이 아니라 판단의 흐름으로 이어지는가.
- 개인적 문장이나 경험이 장식이 아니라 선택의 이유와 연결되는가.
- 사건이나 결과에서 결론으로 바로 뛰지 않고, 글쓴이의 이해와 판단 과정을 독자가 따라갈 수 있는가.
- 개념 설명이 튜토리얼처럼 떠 있지 않고 뒤의 실험·구현 판단을 가능하게 하는가.
- 마지막 문단이 "무엇을 배웠다"보다 "다음 판단 기준이 어떻게 바뀌었는가"로 닫히는가.

### Texture Keeper

polish 과정에서 개발자 관점이 평평해지지 않는지 본다.

- 문제를 보는 방식이 드러난 문장이 단순 요약으로 깎였는가.
- `감각`, `부채`, `스냅샷`, `관심사의 분리`, `중단 기준`처럼 판단 방식을 담은 단어가 사라졌는가.
- 이해, 가정, 비교, 선택 기준이 단순 결과 요약으로 평평해졌는가.
- FE/학습/협업 관점이 generic 기술 설명으로 바뀌었는가.
- 거친 문장이라도 판단 변화나 문제 소유감을 담고 있다면, 없애기 전에 역할을 확인한다.

## 멈춤 신호

- 글이 "무엇을 했다"만 있고 "왜 그걸 문제로 봤는지"가 없다.
- 결과 수치가 있는데 그 수치를 어떻게 읽었는지가 없다.
- 결과에서 결론으로 바로 뛰고, 중간의 이해·가정·비교·선택 기준이 없다.
- 개념 설명이 일반 튜토리얼처럼 떠 있고 뒤의 판단과 연결되지 않는다.
- 기술 선택은 있는데 버린 대안과 제약이 없다.
- 실패나 오독이 삭제되어, 처음부터 잘 판단한 글처럼 보인다.
- polish 후 글쓴이의 문제 해결 방식이 일반 튜토리얼 문체로 사라졌다.
