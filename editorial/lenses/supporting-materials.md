# Supporting Materials

자료 장치는 블로그답게 보이기 위한 장식이 아니라 독자의 판단을 돕는 보조 재료다. 표, 코드, 이미지, 도식, 그래프, 스크린샷은 본문으로 설명하면 길어지는 관계, 비교, 흐름, 전후 차이를 짧게 보여줄 수 있을 때만 쓴다.

이 문서는 새 글쓰기 단계가 아니다. `reader-flow.md`나 `workflow.md`를 보다가 실제 자료 장치 판단이 필요해졌을 때 추가로 여는 처방 문서다.

## 기본 원칙

- 자료 장치는 글감을 대체하지 않는다.
- 자료 장치를 넣기 전에 보호해야 할 문장, 장면, 판단 변화를 먼저 확인한다.
- 자료가 들어가도 본문에는 "왜 보는가"와 "보고 나서 무엇을 판단했는가"가 남아야 한다.
- 자료는 산문 리듬을 나누는 손잡이가 될 수 있지만, 리듬만을 위한 장식이어서는 안 된다.
- 공개 글에는 내부 경로, 비공개 코드, 개인 메모, 원천 자료 화면, 원천 문장을 노출하지 않는다.
- 모바일과 발행 화면에서 깨질 정도로 넓거나 복잡한 자료는 줄이거나 다른 형태로 바꾼다.
- 장치가 글을 더 읽히게 만들었는지와 동시에 더 평평하게 만들었는지 Texture 관점으로 다시 본다.

## Artifact Gap을 풀 자료 고르기

artifact gap이 확인되면 "자료를 넣을까"보다 먼저 "독자가 무엇을 보고 판단할 수 있는가"를 물어야 한다.

- 글의 핵심이 trace, 로그, 실행 흐름, UI 상태, 모델 답변, config, prompt, 실험 결과, 코드 구조 중 하나에 기대고 있다.
- 본문이 "정리했다", "나눴다", "비교했다", "다시 보였다"라고 말하지만 독자가 볼 수 있는 전후 차이, 표, 도식, 스크린샷, 코드 예시가 없다.
- 문장 끝이 비슷한 단정으로 반복되는 구간에서, 실제로는 자료나 캡션이 보여줘야 할 차이를 산문 문장들이 대신 설명하고 있다.
- 독자가 글쓴이의 결론에는 동의할 수 있어도, 실제로 무엇을 보고 그 결론에 도착했는지 복구하기 어렵다.

이 경우 [output-contracts.md](../core/output-contracts.md)의 artifact gap 계약에 따라 직접 삽입, visual candidate, 후보 슬롯, 부적합 판단 중 하나를 남긴다.

## Visual Candidate

artifact gap을 발견해도 하네스는 표나 텍스트 도식으로 쉽게 수렴할 수 있다. 하지만 독자가 실제 상태를 눈으로 확인해야 글의 설득력이 생기는 재료라면 먼저 스크린샷이나 이미지 후보를 검토한다.

우선 검토할 경우:

- 관측 도구, 브라우저 DevTools, 대시보드처럼 UI 자체가 판단 근거인 경우.
- 앱 화면, 모델 답변, 검색 결과, trace tree, latency breakdown처럼 "실제로 이렇게 보였다"가 핵심 장면인 경우.
- 화면, 캡션, 이미지화된 도식이 독자 리듬을 만들 수 있는 재료인데 현재 글은 표와 텍스트 도식만 있는 경우.
- 표로 바꾸면 관계는 정리되지만, 독자가 "저 화면에서 저걸 봤구나"라는 감각을 얻기 어려운 경우.

검토 결과는 최소한 아래 네 가지 중 하나로 남긴다.

| 판단 | 남길 것 |
| --- | --- |
| 바로 넣을 수 있음 | 이미지 경로/삽입 위치/캡션/alt |
| 캡처 필요 | 어떤 화면을 어떤 상태로 찍을지, 가려야 할 정보 |
| 공개 경계 불확실 | 후보 슬롯과 확인할 민감 정보 |
| 시각 자료 부적합 | 왜 표, 텍스트 도식, 코드 예시가 더 나은지 |

## 자료 앞뒤 문장

표, 코드, 이미지, 도식은 혼자 이해되면 좋은 것이 아니라 본문 판단과 연결될 때 힘이 생긴다.

1. **열기**: 이 자료에서 무엇을 봐야 하는지 먼저 말한다.
2. **자료**: 표, 코드, 이미지, 도식, 그래프, 스크린샷을 보여준다.
3. **회수**: 자료를 보고 판단이 어떻게 바뀌었는지 붙인다.

이미지, 스크린샷, 도식에는 사이트 Markdown 계약의 figure caption 형식을 쓴다. 이미지 바로 아래 문단을 `Caption:` 또는 `그림 N.`으로 시작하면 렌더러가 `figcaption`으로 처리한다. `_그림 설명: ..._`처럼 강조문으로 붙이면 캡션이 아니라 본문 강조처럼 보이므로 쓰지 않는다.

캡션은 이미지가 무엇인지 짧게 식별하는 자리다. 독자에게 보이는 작은 라벨에 가깝게 쓰고, 본문에서 해야 할 해석, 판단 변화, 다음 행동까지 캡션에 길게 몰아넣지 않는다. 이미지 없이도 전달되어야 할 내용은 alt에, 이미지에서 무엇을 봐야 하는지는 앞뒤 본문에 둔다.

이미지나 도식을 직접 만들거나 수정했으면 삽입만으로 끝내지 않는다. 반복 카드의 폭과 높이, 좌우 여백, 화살표 기준선, 원/라벨의 중앙 정렬, 하단 보조 문장의 시작점이 한 화면에서 어색하지 않은지 확인한다. 독자가 자료의 의미를 이해하기 전에 배치 불균형을 먼저 느끼면, 자료 장치가 본문 리듬을 돕지 못한다.

자료 뒤 회수 문장은 길게 다시 설명하지 않는다. 보통 아래 중 하나만 맡는다.

- 자료를 보기 전 기대와 달라진 지점.
- 다음 섹션으로 가져갈 판단.
- 공개 경계나 한계 때문에 독자가 조심해서 읽어야 할 조건.
- 이후 실험, 구현, 디버깅에서 바뀐 행동.

## 장치별 기준

- **표**: 여러 항목이 3개 이상의 속성을 가질 때 적합하다.
- **리스트**: 순서가 없거나 단순 항목을 빠르게 보여줄 때 적합하다.
- **번호 리스트**: 실행 순서, 판단 순서, 원인 추적 순서가 중요할 때 적합하다.
- **코드 예시/의사코드**: 로직의 책임, 입력/출력, 프롬프트 구조, 평가 기준을 독자가 봐야 판단할 수 있을 때 적합하다.
- **도식**: 파이프라인, 책임 분리, 데이터 흐름, 원인-결과 경로를 보여줄 때 적합하다.
- **그래프**: 숫자의 크기보다 변화 방향이나 비교 폭이 핵심일 때 적합하다.
- **스크린샷**: 실제 출력, UI 상태, 모델 답변 차이를 독자가 직접 봐야 판단할 수 있을 때 적합하다.
- **이미지/일러스트**: 분위기만 만드는 경우에는 우선순위가 낮다. 글의 핵심 질문이나 증거를 선명하게 할 때만 쓴다.
- **콜아웃/요약 박스**: 운영상 주의점, 실험에서 남은 규칙, 독자가 기억해야 할 제약을 잠깐 멈춰 세울 때 적합하다.

## 후보 표시

필요한 자료나 공개 경계 판단이 남아 있을 때는 해당 위치에 주석으로 후보 슬롯을 남길 수 있다. 슬롯 형식은 [output-contracts.md](../core/output-contracts.md)를 따른다.

이 주석은 편집 단계의 임시 장치다. `content/posts/`로 승격하기 전에는 실제 자료로 바꾸거나 제거한다.

## 멈춤 신호

- 자료가 본문 판단을 돕지 않고 분위기만 만든다.
- 기술 판단을 만든 artifact가 필요한데도 "자료 장치는 기본값이 아니다"라는 이유로 아무 후보도 만들지 않는다.
- trace UI, 제품 화면, 모델 답변처럼 화면성이 강한 재료가 있는데도 스크린샷/이미지 후보를 검토하지 않고 표나 텍스트 도식으로만 끝낸다.
- 스크린샷/이미지를 추천했지만 원고에 후보 슬롯, user-action, public-boundary를 남기지 않고 최종 답변에서만 언급한다.
- 표, 코드, 도식, 스크린샷 앞뒤에 볼 이유와 회수 문장이 없다.
- 자료 장치를 넣으면서 살아 있는 장면, 실패, 판단 변화를 지웠다.
- 시각 자료에 내부 경로, 비공개 코드, 개인 메모, 원천 자료 원문이 노출된다.
