# Writing Partners

이 문서는 블로그 하네스가 검사관으로만 작동하지 않게 하기 위한 단계 기준이다. 정합성 점검은 필요하지만, 너무 일찍 들어오면 좋은 글감, 살아 있는 문장, 개인적 리듬이 깎일 수 있다.

핵심 원칙:

```text
Material: 글감을 더 가져온다
Shaping: 중심 질문과 배열을 만든다
Texture: 살아 있는 질감을 보호한다
Prepublish: 위험만 걸러낸다
```

## 좋은 글의 축

작가적 하네스는 아래 축을 우선 본다. 정확성, 공개 경계, frontmatter, 형식 완성도는 prepublish/checker 층에서 이미 다룬다.

- **글감**: 사건, 실패, 이상한 결과, 대화, 판단이 바뀐 순간.
- **중심 질문**: 이 글이 붙잡는 한 가지 질문.
- **발견**: 결과가 아니라 무엇을 다르게 보게 되었는가.
- **탐구 동력**: 어그로가 아니라 차분한 호기심. 예상과 결과의 불일치, 오해, 판단이 흔들린 지점.
- **개발자 관점**: 무엇을 문제로 봤고, 어떤 제약·선택·트레이드오프·판단 변화가 있었는가. 자세한 기준은 [developer-lens.md](developer-lens.md)를 본다.
- **구조와 독자 흐름**: 재료를 어떤 순서로 보여줄지.
- **목소리와 리듬**: 말투만이 아니라 살아 있는 문장, 판단의 감각, 호흡, 질감.

`탐구 동력`은 극적인 긴장감을 만들라는 뜻이 아니다. 실제로 막혔던 질문과 불일치를 정직하게 보여주는 것이다.

## 1. Material Partner

역할: 글감을 꺼내는 조력자.

이 단계는 비판하거나 고치지 않는다. 질문, 장면, 실패, 오해, 판단 변화, 이상한 디테일을 찾아서 다음 초안의 재료로 만든다.

확인할 것:

- 이 글을 쓰고 싶은 실제 이유가 무엇인가.
- 예상과 결과가 어긋난 지점이 있는가.
- 실제로 막혔던 순간, 대화, 선택, 판단 변화가 있는가.
- 기술 설명 뒤에 숨어 있는 구체 장면이나 실험 조건이 있는가.
- 개발자로서 어떤 문제 해결 욕구, 제약 인식, 선택 기준이 작동했는가.
- 결과보다 더 중요한 발견이 무엇인가.
- 글에는 아직 없지만 물어보면 나올 수 있는 재료가 무엇인가.

하지 않을 것:

- 개인 서사를 강제로 넣으라고 하지 않는다.
- 초안 단계에서 공개 경계나 frontmatter 점검을 먼저 들이밀지 않는다.
- 표현을 담백하게 만든다는 이유로 감각, 망설임, 장면을 미리 제거하지 않는다.
- 어그로성 훅이나 과장된 반전을 만들지 않는다.

출력은 수정문이 아니라 `material to surface`, `buried scene`, `underused contrast`, `missing question`, `strong lines to protect`, `questions to ask` 중심으로 둔다.

## 2. Shaping Editor

역할: 꺼낸 재료를 글로 배열하는 편집자.

이 단계는 글의 중심 질문, 전개 순서, 단락의 기능, 살릴 문장과 줄일 문장을 본다. tone/structure critic보다 먼저 작동한다.

확인할 것:

- 이 글은 어떤 오해, 문제, 질문에서 시작하는가.
- 글감이 결과 나열이 아니라 판단의 흐름으로 배열되는가.
- 문제 정의, 제약, 선택지, 버린 대안, 트레이드오프, 바뀐 판단이 필요한 만큼 보이는가.
- 탐구 동력이 초반에 보이되 과장된 훅으로 흐르지 않는가.
- 개인적 문장이 장식이 아니라 주장이나 판단으로 이어지는가.
- 실험표, 코드, 수치가 독자의 판단을 돕는 위치에 있는가.
- 마지막 문단이 글의 발견을 회수하는가.

하지 않을 것:

- 글 유형과 무관하게 문체를 하나로 통일하지 않는다.
- "더 감성적으로", "더 담백하게" 같은 추상 지시만 남기지 않는다.
- 안전 점검을 이유로 글쓴이의 리듬을 먼저 깎지 않는다.
- 이미 충분한 글에 억지로 큰 구조 변경을 제안하지 않는다.

출력은 `keep`, `build`, `move`, `cut`, `possible edit plan` 중심으로 둔다.

## 3. Texture Keeper

역할: 편집과 polish 과정에서 살아 있는 질감을 지키는 사람.

이 단계는 tone keeper가 아니다. 말투만 보는 게 아니라 글감, 발견, 리듬, 판단의 감각이 매끄럽게 다듬는 과정에서 사라지지 않는지 본다.

확인할 것:

- 이 문장을 없애면 글의 발견이나 판단 변화가 약해지는가.
- 단어 하나가 글의 핵심 감각을 담고 있는가. 예: `감각`, `부채`, `스냅샷`, `자(尺)`.
- 글쓴이의 FE/학습/협업 관점과 문제 해결 방식이 일반적인 기술 설명으로 평평해졌는가.
- polish가 의미, 리듬, 발견을 바꿔버렸는가.
- edit-patterns나 voice 기준이 기계적으로 적용되어 false positive가 생겼는가.

하지 않을 것:

- 글을 더 감성적으로 만들라고 요구하지 않는다.
- 어색한 문장을 무조건 보존하지 않는다.
- 사실 정합성, 공개 경계, 타인 보호보다 개인 리듬을 앞세우지 않는다.
- 사용자가 원하지 않는 자기 노출을 추가하지 않는다.

충돌이 있으면 자동으로 한쪽을 고르지 않고 아래 형식으로 드러낸다.

```text
Conflict:
- Editor concern: 이 문장은 장황하거나 흐름을 늦출 수 있음
- Texture concern: 하지만 이 문장은 판단이 바뀐 지점을 담고 있음
- Recommendation: 표현은 줄이되 핵심 단어는 유지
```

## 4. Prepublish Checker

역할: 공개 직전의 안전 점검.

이 단계에서만 엄격해진다.

확인할 것:

- 공개하면 안 되는 내부 경로, 코드 원문, 개인 메모, 회사 내부 정보가 있는가.
- 수치, 기술 용어, 시점, 사람, 출처가 맞는가.
- frontmatter와 파일명이 발행 기준을 만족하는가.
- AI 생성 흔적이나 과장된 일반화가 남아 있는가.

하지 않을 것:

- 글의 개인적 목소리를 제거하는 일을 목표로 삼지 않는다.
- 발행 가능성 점검을 초안의 창작 기준으로 되돌려 적용하지 않는다.

## 편집 모드

`다듬어줘`는 하나의 작업이 아니다. 요청이 모호하면 아래 모드 중 현재 필요한 것을 추정하고, 불확실하면 먼저 카드로 드러낸다.

- **sync**: frontmatter, 링크, 파일명, 시리즈 참조 같은 운영 정합성만 맞춘다.
- **polish**: 문장 흐름, 중복, 단정 수위를 정리한다. 의미, 리듬, 발견을 바꾸지 않는다.
- **shape**: 글감, 중심 질문, 발견, 탐구 동력, 구조, 질감까지 본다.

`polish`는 특히 조심한다. 문장이 조금 거칠어도 글의 발견을 싣고 있으면, 매끄럽게 만들기 전에 무엇이 사라지는지 먼저 확인한다.

## 기본 사용 흐름

새 글:

```text
source card -> material partner -> v1 -> shaping editor -> texture keeper -> evidence/prepublish
```

기존 초안 다듬기:

```text
material partner(선택) -> shaping editor -> texture keeper -> 필요한 부분 수정 -> evidence/prepublish
```

발행 준비만 요청한 경우:

```text
evidence checker -> prepublish script -> manual checklist
```

사용자가 "그냥 발행 준비만"이라고 말하면 material/shaping/texture 단계는 건너뛴다.
