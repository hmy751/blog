# Reader Flow Lens

가독성은 문장을 쉽게 만드는 일이 아니라, 독자가 글의 판단 흐름을 놓치지 않게 하는 구조 장치다.

이 렌즈는 새 단계나 새 agent가 아니다. `Shaping Editor`가 주로 참고하고, `Texture Keeper`가 polish 과정에서 과한 요약 장치가 글감과 질감을 죽이지 않는지 함께 본다. 필요하면 `Structure Critic`이 구조 리스크를 볼 때 보조로 참고한다.

## 핵심 질문

- 독자가 처음부터 끝까지 정독하지 않아도 글의 질문과 판단 변화를 복구할 수 있는가.
- 소제목만 훑어도 글의 흐름이 보이는가.
- 각 섹션 첫 문장이 그 섹션의 역할과 판단 방향을 여는가.
- 표, 코드 예시, 이미지, 도식, 그래프, 스크린샷이 장식이 아니라 독자의 이해와 판단을 돕는가.
- 숫자, 실험 조건, 비교 결과가 본문에 묻히지 않고 해석으로 이어지는가.
- 요약 장치가 글감을 대체하지 않고, 본문을 읽고 싶게 만드는 손잡이로 작동하는가.

## 좋은 신호

- 소제목이 주제명만 말하지 않고, 그 절에서 드러난 판단이나 변화의 방향을 암시한다.
- 문단 첫 문장이 배경 설명으로만 늦어지지 않고, 독자가 따라갈 중심 생각을 먼저 놓는다.
- 반복되는 실험이나 비교가 같은 리듬으로 배열된다: 처음 기대 -> 실제 관찰 -> 해석 -> 남은 기준.
- 표나 리스트가 긴 산문을 줄이되, 글의 발견이나 판단 변화 문장을 지우지 않는다.
- 숫자는 결과값으로만 놓이지 않고, 그 숫자를 어떻게 읽었는지가 바로 이어진다.
- 도식은 복잡한 파이프라인, 책임 분리, 전후 차이를 빠르게 보여준다.
- 코드 예시나 의사코드는 내부 구현을 노출하지 않으면서 독자가 책임 분리, 입력/출력, 판단 기준을 이해해야 할 때 사용된다.
- 이미지나 스크린샷은 실제 출력 차이, 상태 변화, 독자가 봐야 하는 증거를 보여줄 때 사용된다.

## 소제목과 문단

소제목은 독자가 중간에 다시 진입하는 손잡이다. 설명형 제목이 항상 나쁜 것은 아니지만, 기술 글과 학습/실험 글에서는 다음 질문을 확인한다.

- 이 소제목만 봐도 절의 역할이 보이는가.
- 같은 레벨의 소제목이 비슷한 추상도와 리듬을 갖는가.
- `처음에는`, `하지만`, `그래서`, `결국` 같은 전환이 제목과 본문 첫 문장 중 어디에 있어야 가장 잘 읽히는가.
- 문단 첫 문장이 "무엇을 말하려는 문단인지"를 열고 있는가.
- 한 문단에 서로 다른 생각이 두 개 이상 들어가 있지 않은가.

긴 문단은 무조건 잘라내지 않는다. 다만 한 문단이 실험 조건, 결과, 해석, 다음 판단을 모두 떠안고 있으면 독자가 놓치기 쉽다. 이때는 역할별로 나누거나 표/리스트를 검토한다.

## 표, 코드, 이미지, 도식 사용 기준

표, 코드, 이미지, 도식은 블로그답게 보이기 위한 장식이 아니라 독자의 판단을 돕는 장치다. 본문으로 설명하면 길어지는 관계, 비교, 흐름, 전후 차이를 짧게 보여줄 수 있을 때만 쓴다.

사용 기준:

- **표**: 여러 항목이 3개 이상의 속성을 가질 때 적합하다. 예: `조정한 것 / 처음 기대 / 실제 관찰 / 남은 기준`.
- **리스트**: 순서가 없거나 단순 항목을 빠르게 보여줄 때 적합하다.
- **번호 리스트**: 실행 순서, 판단 순서, 원인 추적 순서가 중요할 때 적합하다.
- **코드 예시/의사코드**: 로직의 책임, 입력/출력, 프롬프트 구조, 평가 기준을 독자가 봐야 판단할 수 있을 때 적합하다. 내부 코드 원문을 공개할 수 없으면 공개 가능한 의사코드나 축약 예시로 바꾼다.
- **도식**: 파이프라인, 책임 분리, 데이터 흐름, 원인-결과 경로를 보여줄 때 적합하다.
- **그래프**: 숫자의 크기보다 변화 방향이나 비교 폭이 핵심일 때 적합하다.
- **스크린샷**: 실제 출력, UI 상태, 모델 답변 차이를 독자가 직접 봐야 판단할 수 있을 때 적합하다.
- **이미지/일러스트**: 분위기만 만드는 경우에는 우선순위가 낮다. 글의 핵심 질문이나 증거를 선명하게 할 때만 쓴다.

사용 전 질문:

- 이 자료가 없으면 독자가 관계나 차이를 놓치기 쉬운가.
- 본문 2~3문단을 표나 도식 하나로 줄일 수 있는가.
- 시각 자료가 글의 핵심 질문을 선명하게 하는가, 아니면 단순히 있어 보이게 하는가.
- 모바일에서 봐도 읽히는가.
- 내부 경로, 비공개 코드, 개인 메모, 원천 자료 화면이 노출되지 않는가.
- 캡션이나 본문 연결 없이도 무엇을 봐야 하는지 알 수 있는가.

## 보조 자료 후보 표시 방식

초안 편집 중 표, 코드, 이미지, 도식, 그래프, 스크린샷이 필요해 보이면 바로 완성본을 강제로 넣지 않는다. 먼저 어떤 장치가 독자 판단을 가장 잘 돕는지 고르고, 필요한 경우 해당 위치에 주석으로 후보 슬롯을 남긴다.

기본 원칙:

- 바로 작성 가능한 Markdown 표, 리스트, 짧은 코드 예시, Mermaid/텍스트 도식은 초안에 직접 넣을 수 있다.
- 실제 이미지, 스크린샷, 그래프, 생성 이미지가 필요하면 원천 자료와 공개 경계를 확인한 뒤 넣는다.
- 이미지 생성이나 편집은 사용자가 요청하거나 프로젝트의 asset 저장 규칙이 있을 때 진행한다.
- 자료가 아직 없거나 스타일 판단이 필요하면 초안에는 주석만 남기고, 사용자가 제공하거나 결정해야 할 것을 분리해서 적는다.

후보 주석 형식:

```md
<!-- reader-flow candidate
type: table | list | code | pseudocode | diagram | graph | screenshot | image
purpose: 독자가 무엇을 더 빨리 판단하게 할 것인가
placement: 어느 문단 뒤에 둘 것인가
source: 필요한 데이터, 출력, 화면, 실험 결과
style: 간단한 Markdown 표, Mermaid 도식, 공개 가능한 의사코드, 실제 스크린샷, 생성 이미지 등
user-action: 사용자가 제공해야 할 자료나 결정할 스타일
caption: 본문 아래에 붙일 설명
alt: 이미지 없이도 전달되어야 할 내용
public-boundary: 내부 경로, 비공개 코드, 개인 메모, 원천 문장 노출 위험
-->
```

이 주석은 발행 전에는 실제 자료로 바꾸거나 제거한다. 주석 자체가 공개 원고에 남아서는 안 된다.

## 숫자와 비교 배치

숫자는 자랑이나 장식이 아니라 판단의 근거다. 숫자를 쓸 때는 아래 순서를 우선한다.

1. 무엇을 확인하려는 실험인지 먼저 둔다.
2. 조건과 비교 대상을 필요한 만큼만 밝힌다.
3. 숫자를 보여준다.
4. 그 숫자를 어떻게 읽었는지 바로 붙인다.
5. 다음 판단이나 남은 한계로 연결한다.

숫자가 많은 글에서는 산문 안에 모든 값을 묻지 말고, 독자가 비교해야 하는 값만 표나 짧은 리스트로 꺼낸다.

## 파트너별 사용

### Shaping Editor

- 소제목만 읽어도 글의 질문, 실험, 판단 변화가 보이는가.
- 각 섹션 첫 문장이 그 섹션의 역할을 분명히 여는가.
- 반복되는 실험/비교는 같은 리듬으로 배열되어 독자가 차이를 쉽게 볼 수 있는가.
- 항목마다 여러 속성을 비교해야 할 때 표나 리스트가 산문보다 나은가.
- 숫자가 본문에 묻히지 않고, 그 숫자를 어떻게 읽었는지가 바로 이어지는가.
- 이미지, 도식, 그래프, 스크린샷 후보가 있으면 본문 어디에서 독자 판단을 돕는지 설명할 수 있는가.

### Texture Keeper / Polish

- 가독성을 높인다는 이유로 살아 있는 장면, 실패, 판단 변화가 표나 요약으로 대체되지 않았는가.
- 소제목을 결과형으로 바꾸면서 글의 차분한 호기심이 과장된 훅으로 변하지 않았는가.
- 문단을 쪼개는 과정에서 리듬이 지나치게 끊기지 않았는가.
- 표나 리스트가 본문을 평평하게 만들면, 핵심 판단 문장을 표 밖에 남겼는가.

### Structure Critic

- 독자가 글의 순서를 따라가야 판단할 수 있는 부분과, 표/도식으로 먼저 봐도 되는 부분이 구분되어 있는가.
- 구조 문제가 단락 재배열 문제인지, 독자 진입 신호 부족 문제인지 구분한다.

## 멈춤 신호

- "가독성"이라는 이유로 모든 문장을 짧게만 만든다.
- 표를 넣었지만 표 밖에 해석이 없다.
- 이미지나 도식이 본문 주장을 대신하고, 본문에서는 무엇을 봐야 하는지 말하지 않는다.
- 소제목이 전부 주제명이라 훑어도 글의 변화가 보이지 않는다.
- 반대로 소제목이 과장된 결론처럼 보여 본문보다 앞서 단정한다.
- 숫자가 많은데 어떤 숫자가 판단의 근거인지 드러나지 않는다.
- 시각 자료에 내부 경로, 비공개 코드, 개인 메모, 원천 자료 원문이 노출된다.
