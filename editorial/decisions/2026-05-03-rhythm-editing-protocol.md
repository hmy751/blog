# 리듬 편집 프로토콜 보강

작성일: 2026-05-03

## 배경

PI Lab 초안을 가독성 기준으로 다듬는 과정에서 표, 도식, 요약 박스, 소제목 조정은 들어갔지만 글의 호흡은 여전히 어색했다. 특히 리듬 문제를 소제목 종결 방식이나 본문 말투 문제로 좁게 해석하면서, 글 전체의 말투를 전역으로 바꾸는 잘못된 편집이 발생했다.

## 기존 하네스가 놓친 문제

`reader-flow-lens`와 `writing-partners`에는 이미 리듬, 호흡, 소제목, 문단 첫 문장, 자료 앞뒤 문장 기준이 있었다. 하지만 실제 작업 절차에서는 리듬 문제가 어느 층위에서 생겼는지 먼저 나누라는 지시가 약했다.

그 결과 다음 혼동이 생겼다.

- 리듬 문제를 표, 도식, 요약 박스 같은 장치 부족으로만 봄.
- 소제목의 `추상도와 리듬`을 맞추라는 기준을 종결어미까지 통일하라는 뜻으로 오해함.
- 본문 존댓말 자체가 문제가 아닌데 전체 문체를 했다체로 바꾸는 식의 전역 수정으로 반응함.
- reader-flow 편집 뒤 Texture Keeper 관점으로 글이 평평해졌는지 충분히 되보지 않음.

## 결정

`reader-flow-lens`에 리듬 문제를 구조, 문단, 전환, 소제목, 문장 끝, 자료 배치의 층위로 나누는 절차를 추가한다.

`writing-partners`와 `blog-write` skill에는 리듬/호흡 요청을 받았을 때 곧바로 문체나 소제목을 전역 변경하지 말고, 문제 층위와 수정 범위를 먼저 정하라는 기준을 추가한다.

active guide에는 특정 임시 참고 자료 이름이나 파일 위치를 남기지 않는다. 참고 자료는 기준을 만드는 배경이 될 수 있지만, 반복 실행되는 하네스는 일반화된 판단 기준만 가진다.

## 적용 범위

- `리듬`, `호흡`, `뭔가 이상함`, `막힘`, `단조로움` 같은 요청.
- 가독성 편집 후 글이 보고서, 체크리스트, 하네스 출력처럼 평평해진 경우.
- 소제목, 전환 문장, 문장 끝, 표/도식 배치가 글의 흐름을 방해하는 경우.

## 비목표

- 모든 글의 문체를 하나로 통일하지 않는다.
- 소제목을 항상 명사형이나 질문형으로 바꾸지 않는다.
- 산문 사이에 장치를 일정 간격으로 강제 삽입하지 않는다.
- 임시 참고 자료의 디자인이나 구성 패턴을 그대로 복제하지 않는다.

## 후속 점검

- PI Lab 02, 03 초안을 다시 다듬을 때 먼저 리듬 층위를 진단한 뒤 수정한다.
- 편집 후 `Texture Keeper` 관점으로 의미, 발견, 판단 변화가 평평해지지 않았는지 확인한다.
