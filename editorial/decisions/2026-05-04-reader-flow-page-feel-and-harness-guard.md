# Reader Flow 목적 재정의와 하네스 구조 가드

작성일: 2026-05-04
상태: accepted

## 배경

Reader Flow는 좋은 글처럼 읽히는 감각을 블로그 하네스 안에 넣기 위해 만들어졌다. 그 감각에는 판단 흐름뿐 아니라 첫 화면의 밀도, 문단 호흡, 표현 리듬, 소제목의 눈길, 스크롤 리듬, 자료 배치, 글쓴이의 판단이 살아 있는지가 함께 포함된다.

하지만 이후 적용 과정에서 Reader Flow가 좁게 해석되는 문제가 반복되었다. 조심하면 산문 polish에 머물렀고, 적극적으로 적용하면 표, 도식, 요약 박스, 문단 호흡 같은 장치가 기계적으로 들어갔다. 대화 중 든 예시들은 의도를 설명하기 위한 임시 재료였고, 하네스가 계속 의존해야 하는 자료가 아니었다. 기존 기록은 이를 특정 참고 자료의 패턴이나 "판단 흐름 장치"로 좁게 읽히기 쉬웠다.

또 한 가지 문제는 사용자의 즉시 지시가 강할 때 AI가 기존 하네스 구조를 충분히 고려하지 않고 바로 해당 지시를 실행하려는 경향이었다. 이 경우 `Material -> Shaping -> Texture -> Prepublish` 구조가 흐려지고, 특정 렌즈나 프로토콜이 전체 편집 권한을 가져갈 수 있다.

## 기존 하네스가 놓친 문제

- Reader Flow의 출발점인 "좋은 글처럼 읽히는 감각"이 문서 상단에 충분히 드러나지 않았다.
- 판단 흐름, 표/도식, 자료 앞뒤 문장 기준이 강해져 정보 배치 최적화나 자료 장치 추가로 오해되기 쉬웠다.
- 프로토콜을 만들면 실행 가능성은 좋아지지만, 다시 진단 기록 양식처럼 작동할 위험이 있었다.
- 사용자 지시와 기존 하네스 구조가 충돌할 때 이를 감지하고 드러내는 규칙이 약했다.
- local Claude/Codex agent가 사용자 지시를 기존 단계 책임보다 우선하는 상황을 별도로 감지하지 않았다.

## 결정

`editorial/lenses/reader-flow.md`의 목적을 넓힌다.

- Reader Flow를 "판단 흐름을 놓치지 않게 하는 구조 장치"가 아니라, 좋은 기술 글처럼 읽히는 표면감과 판단 흐름을 함께 보는 렌즈로 설명한다.
- 페이지 감각, 문단 호흡, 표현 리듬, 스캔 경로, 정보 배치, 판단 흐름을 함께 본다.
- 특정 글이나 임시 자료는 복제 대상이 아니라 읽힘의 효과를 설명하는 예시일 뿐이라고 명시한다.
- 결과는 진단 기록이 아니라 가장 작은 edit move로 이어져야 한다고 정리한다.

기존 하네스 구조도 함께 보호한다.

- `editorial/core/workflow.md`는 글을 키우는 단계 모델과 편집 기준으로 유지하고, 운영 가드 섹션은 두지 않는다.
- `editorial/README.md`, `editorial/core/workflow.md`, 프로젝트 가이드의 Reader Flow 설명은 새 목적에 맞게 좁게 갱신한다.
- `.claude/skills/blog-write/SKILL.md`에 사용자의 하네스 관련 지시를 곧장 새 규칙이나 단계로 승격하지 말고 기존 구조와 충돌 여부를 먼저 확인하라는 가드를 추가한다.
- `blog-shaping-editor`, `blog-texture-keeper`, `blog-structure-critic`의 Claude/Codex 정의에 사용자 지시나 특정 렌즈가 기존 하네스 흐름을 우회하는지 감지하는 항목을 추가한다.

## 비목표

- Reader Flow를 새 독립 단계나 새 프로토콜로 만들지 않는다.
- 모든 글에 페이지 감각 점검표를 강제하지 않는다.
- 특정 글의 디자인, 히어로 이미지, 도식, 표 구성을 복제하지 않는다.
- 사용자 지시를 무시하는 방어적 하네스로 만들지 않는다.
- Texture 보호를 이유로 공개 경계, 사실 정합성, 발행 안전을 약화하지 않는다.

## 적용 범위

- `가독성`, `읽기 좋게`, `reader-flow`, `좋은 글처럼`, `호흡`, `표면감`, `스크롤 리듬` 요청
- 하네스 자체를 수정하는 요청
- 사용자 지시가 특정 렌즈, agent, 구조, 자료 장치를 강하게 요구하는 요청
- PI Lab 글처럼 기존 발행글, 포트폴리오 신호, 공개 경계가 함께 걸리는 글

## 후속 점검

- Reader Flow 적용 결과가 산문 polish에 머물지 않는가.
- Reader Flow 적용 결과가 표, 도식, 요약 박스의 기계적 삽입으로 흐르지 않는가.
- 좋은 글처럼 읽히는 표면감이 실제 문단, 소제목, 표현 호흡, 정보 위치 수정으로 이어지는가.
- 사용자 지시와 기존 하네스 구조가 충돌할 때 충돌이 숨겨지지 않는가.
- Claude/Codex agent 정의가 같은 의미를 유지하는가.
