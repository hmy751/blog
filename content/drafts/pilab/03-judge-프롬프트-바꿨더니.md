# 멀티모달 RAG에서 Judge 프롬프트만 바꿨더니 점수가 +45%

비디오를 받아서 음성을 전사(STT)하고, 프레임을 뽑아 비전 모델로 분석하고, 둘을 합쳐서 임베딩 → 검색하는 멀티모달 RAG를 만들었다. 임베딩 모델을 바꿔서 검색 정밀도가 3배 올랐는데 답변 점수는 따라오지 않았고, "검색을 더 파야 한다"고 판단했다. 결과적으로 틀린 판단이었다 — 문제는 검색이 아니라 평가자(Judge) 쪽에 있었다.

## 멀티모달 RAG 어떻게 만들었나

텍스트 RAG는 PDF나 웹 문서 같은 텍스트만 다루는데, 이번 미션은 비디오가 입력이었다. 비디오에서 정보가 있는 곳은 두 군데 — 사람이 말하는 **음성**과 화면에 보이는 **장면**. 둘 다 검색 가능한 형태로 만들어야 했다.

- **음성**: Whisper로 STT를 돌려 텍스트로 전사
- **비전**: 일정 간격으로 프레임을 뽑아 LLaVA 같은 비전 LLM에 넣고, "이 프레임에 무엇이 보이는가"를 텍스트로 묘사
- **결합**: 같은 시간 구간의 음성 전사와 프레임 묘사를 하나의 청크로 묶어 임베딩
- **검색**: 질문이 들어오면 질문 임베딩과 청크 임베딩의 유사도로 가장 관련 있는 구간을 찾고, 그 청크를 근거로 LLM이 답변

구조가 텍스트 RAG와 크게 다른 점은 **LLM이 파이프라인에 여러 층으로 쌓인다는 것**이다. 음성 전사도 LLM(Whisper), 프레임 묘사도 LLM(LLaVA), 답변 생성도 LLM, 그리고 뒤에서 볼 평가도 LLM(Judge). 각 층이 실패하는 방식이 달라서, 어느 층이 문제인지 분리해서 볼 수 있어야 개선이 성립한다.

## 평가 지표 네 가지와 LLM-as-Judge

평가 지표는 네 가지로 잡았다.

- **WER** (Word Error Rate) — 음성 전사 품질. 정답 전사와 얼마나 일치하는가.
- **AR** (Answer Relevance) — 답변이 질문과 얼마나 관련 있는가.
- **GR** (Groundedness) — 답변이 검색된 청크에 근거하는가 (환각 여부).
- **RP** (Retrieval Precision) — 검색이 질문과 관련 있는 청크를 가져왔는가.

AR, GR, RP는 "답변이 얼마나 좋은가"를 묻는 지표라 객관 정답이 없다. 사람이 매번 채점할 수도 없으니 **LLM-as-Judge** 방식을 썼다. 다른 LLM에게 "이 답변이 얼마나 관련 있나"를 0~1로 채점하게 하는 방식이다. 어쩔 수 없는 선택이지만, 대신 "Judge 자체가 제대로 채점하는가"를 별도로 검증해야 한다는 부채가 생긴다. 이 부채가 나중에 터진다.

여섯 세트 실험을 돌렸다.

## 임베딩 모델 교체로 검색 정밀도 3배

기존 임베딩 모델 `nomic-embed-text`가 한국어에 약하다는 이야기가 있어서, `bge-m3`로 교체해봤다.

| 지표              | nomic-embed-text | bge-m3   | 개선폭          |
| ----------------- | ---------------- | -------- | --------------- |
| 검색 정밀도 (RP)  | 0.09             | 0.26     | **+189% (3배)** |
| 답변 관련성 (AR)  | 0.49             | 0.53     | +8%             |
| 근거 일치도 (GR)  | 0.80             | 0.89     | +11%            |

모델 이름 문자열을 바꾼 것만으로 검색 정밀도가 3배. 단일 변경 중 가장 큰 임팩트였다.

## 검색 개선 대비 답변 점수가 안 따라오는 문제

검색이 3배 좋아졌는데 답변 관련성은 +8%, 근거 일치도는 +11%에 그쳤다. 검색이 개선된 폭에 비하면 답변 쪽은 거의 안 움직인 셈이다.

처음엔 "검색을 더 파야 한다"고 생각했다. top_k를 줄여볼까, rerank를 도입할까. 그런데 멀티모달에서 답변 점수는 한 층만의 문제가 아니다. 비전 묘사가 제대로 반영됐는지, 전사가 정확했는지, 답변 생성이 검색된 청크를 잘 썼는지 — 그리고 **점수를 매기는 Judge가 제대로 채점하는지**까지. 층이 많은 만큼 "왜 답변 점수가 안 올랐는가"의 설명 후보도 많다. 그러다 누군가 "검색은 그대로 두고, Judge 프롬프트만 새로 짜보면?"이라고 제안했다.

## Judge 프롬프트만 교체한 결과

기존 Judge 프롬프트는 3줄짜리 — "관련성을 0.0~1.0으로 평가하세요" 정도. 새 프롬프트는 역할을 부여하고, 5단계 채점 기준을 명시하고, 타임스탬프 같은 예외 규칙을 추가했다.

검색 결과는 그대로 두고 Judge만 바꿨다.

| 지표 | Judge v1 | Judge v2 | 개선폭   |
| ---- | -------- | -------- | -------- |
| AR   | 0.55     | 0.80     | **+45%** |
| RP   | 0.33     | 0.48     | **+44%** |

같은 검색 결과, 같은 답변인데 점수가 거의 절반이 올랐다. 검색을 고쳐야 한다고 판단했던 근거 중 상당 부분이, 사실은 Judge가 제대로 채점을 못 하고 있었던 거였다. 검색이 멀쩡한 부분까지 낮게 채점한 셈이다.

## Judge 모델 자체의 영향

같은 검색 결과, 같은 질문, 같은 Judge 프롬프트에 모델만 바꿔봤다.

| 임베딩 | Judge 모델  | RP        |
| ------ | ----------- | --------- |
| bge-m3 | llama3.1    | 0.381     |
| bge-m3 | gpt-4o-mini | **0.571** |

같은 데이터에 RP가 50% 차이. **평가자는 실험 변수가 아니라 자(尺)였다.** 실험마다 자를 바꾸면 앞뒤 측정값을 비교할 수 없다. 여기서 확인한 건 "실험 간에 Judge를 통일하지 않으면 비교 자체가 성립하지 않는다"였다.

## 각 층의 언어·용량 한계

비슷한 시기에 LLaVA 비전 모델에 한국어로 답하라고 시켰더니 "hello hello hello", "I am number one" 같은 응답이 돌아왔다. 모델이 한국어를 지원하지 않는데 프롬프트로 한국어를 요구한 거다. 2편에서 프롬프트로 검색의 결함을 메우는 접근이 안 됐던 것과 비슷한 구조인데, 여기선 모델 능력 밖의 걸 프롬프트로 요구하면 결과가 오히려 더 나빠졌다.

LLaVA만 문제가 아니었다. Whisper도 영어 대비 한국어 전사 정확도가 낮았고, Ollama로 띄운 로컬 LLM은 긴 프롬프트를 제대로 못 따라가서 프롬프트를 더 짧고 간결하게 바꿨을 때 답변 품질이 오히려 올랐다. **"멀티모달 RAG에서 한국어 지원"은 한 덩어리가 아니라 층마다 따로 있었다.** 각 층의 언어 지원 수준과 용량(긴 프롬프트 이해 능력)을 따로 확인해야 했다.

프롬프트 엔지니어링을 잘한다는 게 모든 모델에서 똑같이 좋다는 뜻이 아니었다. 모델 체급에 맞는 프롬프트가 따로 있었다.

## 이후 바뀐 작업 순서

이후로는 검색을 고치기 전에 Judge가 일관된지부터 확인했다. 새 실험 전에 평가 프롬프트 버전을 맞추고, Judge 모델이 바뀌었으면 이전 결과와 직접 비교하지 않았다. 1편에서 "변수를 통제해야 결과를 해석할 수 있다"는 이야기를 썼는데, 평가 시스템도 통제해야 하는 변수 중 하나였다.
